## On the project

dossier's architecture continues to pay off. The gateway-as-sole-choke-point
pattern meant that multi-project fronting was primarily a routing and registry
problem â€” no provenance logic changed. Every mutation still goes through a
single `RegistaGateway` per project, with the server-resolved `Actor` injected.
The `GatewayRegistry` is a thin cache that mirrors agent-notes' `face_factory`
pattern, adapted for a multi-threaded web server (with double-checked locking).

The slugâ†”schema-name mapping (hyphensâ†”underscores) is the same rule agent-notes
uses, so the two faces address the same schema for the same software-project.
This is a small but load-bearing convention â€” if either face drifts, the human
and agent would address different schemas for the same project name.

The biggest architectural question this session surfaced is the
`known_projects` allowlist. The plan says "MVP: any authenticated human sees
all" â€” but "all" must mean "all configured projects," not "all schemas in the
database." The first implementation got this wrong: `GatewayRegistry.get()`
would lazily build a gateway for any valid-looking schema name, meaning an
authenticated user could probe arbitrary schemas and dossier would sign events
into them with its HMAC key. Both adversarial reviewers caught this. The fix
is simple (check `_known_projects` before building), but the fact that it was
masked by the test fixture (which used `_settings=None`, making `get()` raise
`KeyError` for unknown projects) is a lesson: the test fixture's production-
divergent setup hid a production-only bug.

## On the work done

**Confident in:**
- The `known_projects` enforcement. After the fix, `get()` raises `KeyError`
  for any project not in the configured set. This is the allowlist gate that
  prevents unauthorised schema access. The test `test_known_projects_enforced`
  verifies this with a separate test client that has `settings=None` (matching
  the test fixture) AND the production path (which has `settings` set) â€” wait,
  actually it doesn't test the production path because the fixture uses
  `InMemoryRegista` without settings. The production `_build` path is still
  untested. This is a gap (see below).
- The thread-safe double-checked locking. The `threading.Lock` around the
  build path prevents two concurrent requests from creating duplicate
  `Regista` instances for the same project.
- The slugâ†”schema mapping. `slug_to_project` delegates to regista's
  `validate_project_name`, so the mapping is as correct as regista's own
  validation. The `project_to_slug` reverse is a simple `_`â†’`-` replace.
- The cross-project landing page. It fans out across known projects and is
  resilient to individual project failures (wrapped in try/except with
  logging).

**Want a second pair of eyes on:**
- The `resolve_gateway` catching `(KeyError, RegistaError)`. In production,
  if `registry.get()` calls `_build()` which calls `Regista.__init__()`, and
  the schema doesn't exist, regista raises `RegistaError(DB_NOT_FOUND)`. This
  is caught and returns a 404. But if the database is *down*, it also raises
  `RegistaError` (or `OperationalError`), and we return a 404 instead of a
  503. This is arguably correct (the project is "not found" from the user's
  perspective) but could mask infrastructure issues.
- The `current_project` fix. I changed all route handlers to set
  `ctx["current_project"] = slug_to_project(project)` so it holds the
  canonical schema name (e.g., `dossier_test`), matching what
  `registry.list_projects()` returns. The template compares
  `current_project == p` where `p` is a canonical name. This should work,
  but the switcher trigger displays `project_to_slug(current_project)` which
  adds a layer of conversion. If `slug_to_project` and `project_to_slug`
  are not perfect inverses for some edge case, the display would be wrong.
- The `DOSSIER_PROJECTS` validation in the CLI. I validate each entry with
  `slug_to_project(p.replace("-", "_") if "-" in p else p)` â€” this is
  awkward. The entries in `DOSSIER_PROJECTS` are schema names (with
  underscores), not slugs (with hyphens). But the validation converts them
  to slugs first and then back. It would be cleaner to validate directly
  with `validate_project_name`.

**Awkward:**
- The `_build` path (production lazy-gateway creation) is completely
  untested. All test fixtures use `GatewayRegistry(known_projects=[...])`
  without `settings=`, so `_settings=None` and `get()` raises `KeyError` for
  unknown projects. In production, `settings=` is always provided. The fix
  for the known_projects enforcement means `get()` now checks
  `_known_projects` *before* checking `_settings`, so unknown projects
  raise `KeyError` regardless of whether settings are provided. But the
  happy path (known project + settings provided + lazy build) is untested.
  This would require a Postgres-backed test or a mock Regista factory.
- The CSS is in the vendored `tokens.css` file. The file header says
  "VENDORED FROM patina â€” do not edit here," but all the existing `ds-`
  component CSS is also in this file. The new `.ds-projswitch` and `.ds-check`
  classes follow the same pattern. This is a pre-existing convention issue,
  not something I introduced, but it's worth noting.
- The `regista._connection.validate_project_name` import in `slug_to_project`
  is a private API import. If regista reorganizes its modules, this breaks.
  There's no public API for this on regista's facade.

## Gaps to flag

- **`_build` path untested:** The production lazy-gateway creation path is
  completely untested. All fixtures use `settings=None`, so `_build` is never
  called. The `known_projects` enforcement fix means unknown projects raise
  `KeyError` regardless, but the happy path (known project + lazy build) has
  no test coverage. `src/dossier/multi.py:_build`.
- **`resolve_gateway` catches `RegistaError` broadly:** DB-down and
  schema-missing both return 404. A 503 for DB-down would be more accurate.
  `src/dossier/app.py:resolve_gateway`.
- **`DOSSIER_PROJECTS` validation is awkward:** The validation in `_cmd_serve`
  converts schema names to slugs and back. Should validate directly with
  `validate_project_name`. `src/dossier/cli.py:_cmd_serve`.
- **Private API import:** `slug_to_project` imports
  `regista._connection.validate_project_name`. If regista reorganizes, this
  breaks. `src/dossier/multi.py:slug_to_project`.
- **No lifespan handler:** `close_all()` is called via `try/finally` in
  `_cmd_serve`, but there's no FastAPI lifespan handler. If the app is used
  outside the CLI (e.g., embedded in another ASGI server), `close_all` won't
  be called. `src/dossier/cli.py:_cmd_serve`.
- **regista migration 36 pending:** Blocks `agent-notes breadcrumb reconcile
  --apply`. WI-001 and WI-010 are resolved in git but not in the DB.
  `regista/src/regista/_migrations.py`.
- **Cross-project references not rendered:** Plan 011 WI-4 (render regista
  value-references as navigable links) is not implemented. The landing page
  shows project names and counts but no cross-project item links.
  `src/dossier/templates/landing.html`.
