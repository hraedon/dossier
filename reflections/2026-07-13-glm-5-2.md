## On the project

dossier is maturing well. The separation of concerns is clean: the gateway is the single mutation seam, the assurance module is a pure function of the event log, and the health check follows the suite-conformant shape. The codebase is small enough (~620-line gateway, ~90-line assurance) that fixes are surgical. The test suite (520 tests) is thorough and runs fast (~25s), which makes the TDD loop tight.

The biggest architectural tension remains WI-012 (assurance computation is home-grown in dossier rather than delegated to regista). The fix for WI-014 makes the home-grown path safer, but the root cause — dossier is making a provenance judgment that arguably belongs in the engine — is still open. The threading.Lock approach for WI-011 is a pragmatic single-process fix, but the multi-process gap is real and documented. A regista-side sequence or advisory lock is the correct long-term solution.

## On the work done

**What went well:** The WI-014 fix was a one-line logic inversion (`reviewer_lineage in author_lineages` → `reviewer_lineage not in author_lineages` in the cross-lineage condition) that cleanly closes the fail-open. The WI-019 fix was trivially correct. The WI-011 fix (threading.Lock + max-sequence) is structurally sound for single-process deployments. The adversarial review process caught real issues — the type-coercion bug (author lineages were `str()`-coerced but reviewer side wasn't) was a genuine over-claim avenue that I'd have missed without the review.

**What I'm less confident in:** The `_existing_display_keys` scan is O(n) per create, which is strictly worse than the old `_count_work_items` (it deserializes custom_fields for every item). This is already tracked as WI-018, but the fix made it slightly worse. The concurrent test uses `threading.Barrier` with InMemoryRegista, which may pass due to GIL serialization rather than truly exercising the lock. The WI-019 test using `MagicMock(spec=GatewayRegistry)` is a good unit test but doesn't test through the actual HTTP endpoint.

**The duplicate code path in `create_issue`** (mint path returns inline inside the `with` block, non-mint path has a separate return) is a minor maintenance trap. I considered refactoring to a single return but decided the duplication was small enough to not warrant the added complexity.

## On what remains

- **WI-012 (medium):** Assurance level computation should be delegated to regista. The home-grown path is now safer but still architecturally wrong — dossier is making a provenance judgment.
- **WI-017 (medium):** authz.py is flat-open — every authenticated actor can read every project. The ACL infrastructure exists (Plan 014 WI-1.2), but the default is still "open."
- **WI-016 (medium):** app.py is a 1424-line monolith. The administration, provenance, and operations modules have been extracted, but the main routes file is still too large.
- **WI-018 (medium):** Several read paths are O(n). The `_existing_display_keys` scan I added makes this slightly worse.
- **WI-020 (high):** Plan 023 central k8s deployment — large infra task, not code.
- **Multi-process display key race:** The `threading.Lock` only works within a single process. Multi-worker deployments (gunicorn with `--workers > 1`) still have the race. This needs a regista-side sequence or advisory lock.

## Gaps to flag

- `health.py:309` — `_project_access_check` passes `settings.session_secret` as the `group_claim_key` to `load_project_access_policy`. This looks like a pre-existing bug (session secret is for cookie signing, not group claim blinding). Flagged by the GLM reviewer but not part of this session's scope.
- `gateway.py:118-130` — The `create_issue` method has duplicated `create_work_item` call paths (mint vs non-mint). If a new parameter is added to `create_work_item`, only one path may be updated.
- `assurance.py:59` — Agent `accept` with cross-lineage lineage sets `has_cross_lineage_review = True`. This conflates a final gate (`accept`) with an adversarial review (`adversarial_pass`). May over-claim independence. Pre-existing, not introduced by this fix.
- The `_existing_display_keys` scan in `gateway.py` is O(n) per create and deserializes custom_fields for every work item. This is a performance regression from the old count-based approach, though both were O(n). WI-018 tracks the broader read-path performance issue.
- The concurrent test (`test_display_key_concurrent_no_duplicates`) may pass due to GIL serialization of InMemoryRegista rather than truly exercising the lock. A more rigorous test would use a mock that introduces controlled delays, but this is adequate for MVP.
