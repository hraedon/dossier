## On the project

dossier continues to feel like the cleanest of the suite faces — a thin server-rendered shell over regista, with provenance as the organizing principle. The `_build` lazy-construction pattern in `multi.py` is elegant for the multi-project case but is exactly where secrets resolution belongs, and it was missing it entirely. The pattern of mirroring agent-notes' hardened modules (secrets, soon the suite env loader) rather than re-deriving them is the right call: dossier and agent-notes are siblings over the same regista, and custody contracts should be identical between them. The AGENTS.md rule "regista is the source of truth, dossier caches reads but never owns writes" held cleanly here — secrets.py only resolves *into* regista's providers, never redefines them.

## On the work done

The agent-notes reference implementation was a gift: two prior adversarial rounds had already hardened the silent-literal-fallback guard, the `from None` cause-suppression, and the atexit safety net. Porting it was mostly mechanical (rename temp prefix, drop the agent-notes-specific docstring framing). What was *not* mechanical was the dossier-specific wiring — and that's where both adversarial rounds earned their keep:

- Round 1 (Kimi) caught that `load_settings` derives `principal_key_dir = Path(hmac_key_path).parent / "principals"`, which for a `vault:` ref becomes `Path("vault:...")` → CWD. That's a private-key leak path I would have missed. Also caught the resolved-DSN-bound-to-a-local concern (traceback locals echo passwords) and the `_build`/`_cmd_init` failure-path temp-file leaks.
- Round 2 (GLM) caught the subtler one: if `register_workflow()` raises *after* `Regista()` succeeded, the connection pool leaks even though the temp file gets scrubbed. Also the `os.write` failure orphaning the mkstemp file, and the `file:`-ref `~`-expansion inconsistency between `config.py` and `materialize_key_manifest`.

The two rounds were complementary — round 1 was breadth (did you wire it everywhere?), round 2 was depth (are the failure paths sound?). Worth doing both for security-sensitive code.

Confident in: the resolver module itself (faithful port), the silent-fallback guard, the temp-file lifecycle. Less confident / documented as known-minor: the `del resolved_dsn` in `cli.py` only covers `register_workflow` failure, not `Regista()` constructor failure (the constructor is self-cleaning per regista, so the window is narrow). `_ALL_PROVIDERS` is hard-coded and diverges from regista if regista adds providers (e.g. `windows`/`akv`) — left as-is to match agent-notes.

## Gaps to flag

- `src/dossier/secrets.py:73` — `_ALL_PROVIDERS` is a hard-coded copy of regista's provider registry. If regista registers `windows`/`akv`/future providers, dossier's `is_backend_ref` will treat those refs as literal and hand them to psycopg as bogus DSNs. agent-notes has the same gap; fixing it in dossier would be a chance to fix both.
- `src/dossier/cli.py:~100` — `del resolved_dsn` only scrubs the local on the `register_workflow` failure path, not the `Regista()` constructor failure path. Narrow window (regista self-cleans), but a plaintext DSN could appear in traceback frame locals if the constructor raises. Documented; not fixed.
- `src/dossier/health.py:59-64` — pre-existing: the regista-reachability check echoes `str(exc)[:200]` into the health detail. With resolved secrets, a connection error could surface host/port from the DSN. The new `_secrets_backend_checks` is type-only; this older handler is not.
- `_provider_prefix` is private (`_`-prefixed) but I import it nowhere now (round-1 fix removed that); the `is_backend_ref` + `startswith("file:")` pattern in `config.py` is the public surface. If someone later needs the prefix, prefer adding a public helper over re-importing the private one.
