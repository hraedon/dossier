# Session Reflection — 2026-07-03

**Work summary:** Performed a three-lineage adversarial review (Kimi, Nemotron, Minimax) of the `plan/auth-ownership-planning` feature branch (14 commits, 75 files, ~8,331 lines). Fixed 12 security/correctness issues identified by the review. Merged to main, scrubbed git history of work-domain identifiers, and flipped the GitHub repo from private to public.

## On the project

Dossier is shaping up well as a provenance instrument. The separation between dossier (UI, auth, workflow definition) and regista (state machine, event log, signing) is clean. The gateway pattern — where every mutation goes through `RegistaGateway` with a server-resolved `Actor` — is the right architecture. The LDAP search-then-bind with objectGUID-keyed identity is correct and well-tested.

The main tension I see is the gap between the "v1 flat-open knowingly" authz seam and the provenance posture the project aspires to. `can_read_project` returning `True` for all authenticated users is documented and intentional, but it means the project is a provenance instrument without access control — anyone who can log in can see and mutate every project. This is fine for a homelab MVP but needs to be addressed before any multi-team use.

The publication-review infrastructure is solid — the CI identifier-gate, the manual checklist, and the history scrub process are all well-documented and executable. The fact that the repo had real identifiers in its history (mvmpostgres01, hraedon.com, author emails) and a clean scrub was possible speaks to the discipline of the process.

## On the work done

The adversarial review was thorough. Three independent reviewers found overlapping but distinct issues, which is exactly what cross-lineage review is for. The Kimi reviewer focused on auth internals and found the `set_owner_route` privilege escalation (CRITICAL) and the scrypt N floor gap. Nemotron found the `verify_event` unregistered-signer issue and the throttle evict-locked-records bug. Minimax caught the CI identifier-gate false positive and the Dockerfile HEALTHCHECK timeout.

The 12 fixes I made are all narrowly scoped and well-tested:
- `require_admin` on `set_owner_route` — the most important fix, a real privilege escalation
- `verify_event` forcing `verified=False` for unregistered keys — provenance correctness
- Scrypt N floor warning + upper bound cap — defense in depth
- Throttle lockout extension + locked-record eviction protection — correctness
- Atomic `add_user` write — race condition fix
- Key-id entropy strengthening — defense in depth
- `principals.get_active` expiry check — correctness
- `_cn_from_dn` always using `ldap3.utils.dn.parse_dn` — correctness
- Dockerfile HEALTHCHECK timeout — reliability
- `.gitignore` additions — safety
- CI broad-scan exclusion fix — CI correctness

I'm confident in all 12 fixes. The tests (227 pass, 5 skip) and lint/typecheck are clean. The history scrub was clean — only detection patterns remain in the code, not actual identifiers.

## Gaps to flag

- `app.py:769,899` — Key rotation and enrollment use `secrets.token_bytes(32)` as Ed25519 public keys. These are not valid Ed25519 keys and signature verification will fail. Documented as v1 stubs but if deployed as-is, provenance is broken.
- `app.py:970-1003` — Break-glass action validates dual-control fields but performs no action. Operators may believe the control works.
- `gateway.py:350-381` — Display key minting has a TOCTOU race (count then create). Two concurrent creates can produce duplicate keys. Tracked as WI-011.
- `authz.py:22` — `can_read_project` returns `True` for all authenticated users. Documented as "v1 flat-open knowingly" but means no project isolation.
- `assurance.py:26-67` — Assurance level computation has a logic issue where `accept` by a human actor double-counts (falls through to the `elif`). WI-012 tracks moving this to regista.
- `docker-compose.yml:15-16` — Dev passwords (`dossier:dossier`) are committed. Acceptable for dev compose but should use `.env` file references.
- `tests/test_e2e_postgres.py:14` — Hardcoded DSN with credentials. Should use env var with default.
