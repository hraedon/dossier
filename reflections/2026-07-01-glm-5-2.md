## On the project

dossier's architecture continues to pay dividends. The gateway-as-sole-choke-
point pattern meant that display key minting was isolated to `create_issue` in
`gateway.py` — one method, one place where the sequence is derived and the
custom field is injected. No other code needed to know about the minting
logic. The `display_key()` helper in `web.py` is a pure read-side function
that falls back gracefully for pre-WI-006 items.

The cross-repo dependency on regista's canonical workflow is the part that
feels both right and awkward. Right: the workflow is the contract, and a new
custom field is a contract change that belongs in the shared definition.
Awkward: a dossier-specific UI concern (human-friendly keys) is now in a
workflow shared with agent-notes (breadcrumb type). The field is optional and
the comment explains why, but it's a small leak of presentation into the data
contract. If this pattern recurs (more UI-only fields in the canonical
workflow), we should consider a dossier-only overlay or a separate
`ui_metadata` JSON field.

The publication-review document is straightforward but important. The
checklist is concrete enough to be actionable and the "what constitutes a
work-domain identifier" table gives clear examples. The one gap I notice: it
doesn't mention the `reflections/` directory as a vector for work-domain
leakage — I added a step for it, but future reflections should be written with
this in mind.

## On the work done

**Confident in:**
- The sequence-ownership decision. Deriving the sequence from a paginated count
  (a read) and storing it through the regista create event (a write through
  regista) respects "dossier may cache reads but never own writes." No
  dossier-owned counter table. The race condition is real but documented and
  filed (WI-011).
- Project name sanitization. The adversarial review correctly caught that raw
  project names with spaces or special characters would produce broken keys.
  The fix sanitizes to `[A-Z0-9_]` with a `PROJECT` fallback.
- The `cast(Any, ...)` removal. The second review caught that I cargo-culted
  the `cast(Any, ...)` pattern from pre-existing code. Using `QueryPage[WorkItem]`
  as a type annotation instead is strictly better — if regista renames `items`
  or `has_more`, mypy will catch it here.
- The v2 workflow bump. Adding an optional field and bumping the version is
  safe — regista's workflow registry stores versions as composite keys, so v1
  items continue to resolve to v1 transitions.

**Want a second pair of eyes on:**
- The `_count_work_items` pagination. I verified the cursor-based pagination
  contract against both backends (InMemory and the Postgres path), but there's
  no test that creates >1000 items to exercise multi-page counting. An
  off-by-one in cursor handling would go undetected.
- The `display_key()` web helper fallback. For pre-WI-006 items, it returns
  `str(work_item_id)[:8]` — a truncated UUID. In a table full of `DOSSIER-N`
  keys, a single `a1b2c3d4` will look odd. It's the best we can do without a
  migration, but it's a UX wart.
- Cross-face numbering. `_count_work_items` counts ALL work items in the
  workflow, including breadcrumbs from agent-notes. So if agent-notes creates
  a breadcrumb between two dossier creates, the dossier sequence skips a
  number. This is consistent with the "one work-item universe" design but
  could confuse users who expect contiguous `DOSSIER-N` numbering.

**Awkward:**
- The `test_v2_work_item_transitions_correctly` test. The first adversarial
  review flagged that my original "v1 compat" test didn't actually test v1
  compat (it tested v2 items). I renamed it with an honest docstring, but the
  real v1 backward-compat test would require constructing a v1 workflow YAML
  (without `display_key`), registering it, creating an item, then registering
  v2 and transitioning the v1 item. This is a regista-side test, not a dossier
  test — I noted this in the docstring but it's a gap.
- The regista working tree had an uncommitted change in `_contract.py`
  (`_ALLOWED_ENTITY_KINDS` adding `"session"`) that I had to revert. It was
  leftover from a previous session on the `wip/plan-022-p5-entity-kind` branch.
  I checked out `plan-012-regista-worktracking` and the change carried over. Not
  a big deal, but it means the regista working tree state is fragile across
  branch switches.

## Gaps to flag

- **Display key race condition:** Two concurrent creates can mint the same
  `DOSSIER-N` key. Filed as WI-011. `src/dossier/gateway.py:_mint_display_key`.
- **No pagination test for _count_work_items:** The cursor-based pagination
  loop is only exercised with <5 items. An off-by-one in cursor handling
  would go undetected. `src/dossier/gateway.py:_count_work_items`.
- **Cross-face numbering gaps:** Agent-notes breadcrumbs inflate the sequence
  counter, causing non-contiguous dossier numbering. Consistent with the
  shared-universe design but potentially confusing. `src/dossier/gateway.py`.
- **`from_settings` is dead code:** Never called; `_cmd_serve` constructs the
  gateway directly. `src/dossier/gateway.py:from_settings`.
- **`cast()` calls in gateway.py mask regista API changes:** All other
  regista returns are `cast()` because regista has no `py.typed`. If regista
  changes a return shape, the cast will silently accept it. Pre-existing,
  documented in prior reflection. `src/dossier/gateway.py`.
- **regista working tree had uncommitted changes from another branch:** The
  `_contract.py` change (adding `"session"` to `_ALLOWED_ENTITY_KINDS`) was
  left over from `wip/plan-022-p5-entity-kind`. I reverted it. The regista
  team should verify this change is intentional and committed on the correct
  branch. `regista/src/regista/_contract.py`.
