# Session Reflection — 2026-07-03

**Work summary:** Implemented Plans 014 (cross-project human interface) and 015 (key-management UX) in a single commit. Plan 014 delivers the authorization seam (`can_read_project`), cross-project dashboard with open items across all projects, estate-wide search, verified-history view with per-event cryptographic verification badges, review-assurance level computation, and read-oriented affordances for non-authors. Plan 015 delivers the "my signing identity" view (fingerprint, status, dates — no private key material), self-service key rotation, signing history, admin principal roster with enrollment, revocation (the leaver path), and break-glass form. Two independent adversarial reviews (GLM + Kimi) identified critical gaps; twelve fixes were applied before commit. 48 new tests (217 total), ruff clean, mypy strict clean.

## On the project

Dossier is maturing from a single-project MVP into a cross-project human face for the suite's work. The authz seam (Plan 014 WI-1.1) is the load-bearing design choice — it's what makes v1's flat-open-read model a planned increment rather than a retrofit. The fact that every project-accessing route goes through one function, and a test verifies that, gives me confidence that v1.5's enforcement flip is a real flag change, not a hunt-and-peck across the codebase.

The key-management UX (Plan 015) is more aspirational. The plans reference regista Plan 026 (per-actor Ed25519 signing) and Plan 027 (assurance levels) as dependencies, but those aren't fully landed in the regista we're building against. So the implementation has the right *shape* — the right routes, the right templates, the right test coverage — but the crypto layer underneath is stubbed. The adversarial reviewers caught this hard: `secrets.token_bytes(32)` is not a valid Ed25519 public key, and no signed events are written for key lifecycle actions. These are documented as v1 test stubs with TODOs, but they're real gaps between the plan's promises and what shipped.

The patina design system continues to serve well — the templates are legible, the CSS class naming is consistent, and extending the nav and adding new page types was straightforward. The dark-default look is cohesive.

## On the work done

**What went well:** The parallel subagent strategy was effective — two subagents created all 11 templates in parallel while I handled the app.py integration. Two more subagents wrote the test suites in parallel. The adversarial reviews were genuinely valuable: both reviewers independently caught the same critical issues (break-glass is a no-op, key lifecycle doesn't write signed events, multi-project lifecycle only touches the first project), plus several unique findings each. The fixes were mechanical and quick.

**What I'm confident in:** The authz seam is correctly wired and tested. The dashboard, search, and verified-history views work and are XSS-safe. The template structure is clean and matches existing conventions. The test coverage is thorough — 48 new tests covering authz, dashboard, search, assurance levels, key management, CSRF, and auth gating.

**What I'd want a second pair of eyes on:** The assurance level computation in `assurance.py` is home-grown — it reads event metadata and applies Python logic rather than delegating to a regista API (Plan 027). If regista's computation differs on edge cases, the UI will diverge from the cryptographic truth. The `InMemoryPrincipalKeyStore` test double lives in `src/dossier/principals.py` (production source) rather than `tests/` — the reviewers flagged this correctly. The dynamic project discovery (`_discover_from_catalog`) only works for projects whose gateways are already built, which is circular — it can't discover truly new projects without a running gateway to query the catalog through.

## On what remains

**Needed before this can ship:**
1. **Regista migration 38** — the agent-notes CLI can't file breadcrumbs or reconcile work items because the `dossier` and `agent_notes` schemas are missing migration 38. Postgres isn't running in this environment either. This blocks the end skill's breadcrumb filing step.
2. **Real Ed25519 keypair generation** — rotation and enrollment use `secrets.token_bytes(32)` which is not a valid Ed25519 key. The TODO in `app.py` marks this clearly. Production needs the secret backend to generate the keypair and return only the public key.
3. **Key lifecycle events** — rotation, enrollment, revocation, and break-glass should write signed regista events to the audit trail. Currently they only update the principal key store. This is a Plan 015 principle ("every lifecycle action is an attributed, signed event") that's not met.
4. **Break-glass actual action** — the POST route validates inputs but performs no action and writes no event. It's a form-validation stub.
5. **Dynamic project discovery** — the catalog adapter can't discover projects not in the static known set. Needs a way to query the catalog independently (e.g., via a configured default gateway or a direct DB connection).

**Nice to have:**
- Assurance level delegation to regista Plan 027 API (filed as breadcrumb — but couldn't file it because Postgres is down).
- Performance: search and signing-history do unbounded O(projects × items × events) scans. Need pagination and server-side queries.
- `InMemoryPrincipalKeyStore` should move to `tests/` — it's a test double in production source.
- Dashboard uses schema names instead of catalog display names.

## Gaps to flag

- **`app.py` key generation** (`rotate_my_key`, `enroll_principal`): `secrets.token_bytes(32)` is not a valid Ed25519 public key. Marked with TODO but it's the production code path.
- **`app.py` break-glass**: POST validates dual-control but performs no action. Silent success (303 redirect) with nothing recorded.
- **`assurance.py:28-69`**: Home-grown assurance level computation, not delegated to regista. Could diverge from Plan 027 on edge cases.
- **`multi.py:100-128`**: Dynamic project discovery is circular — only discovers projects whose gateways are already built.
- **`principals.py`**: Test double in production source (`src/dossier/`), not `tests/`. Gateway falls back to it when `_principal_store` is set — accidental injection could bypass the real registry.
- **`app.py` signing-history**: Matches `actor_id`, not cryptographic signer (`principal_id`). Delegated signatures or payload-level actor spoofing will be misattributed.
- **Multi-project key lifecycle**: Fixed to iterate all projects (was first-only), but no test verifies consistency across multiple projects. The existing tests inject one project only.
- **Postgres not running**: Could not file breadcrumbs or reconcile work items via agent-notes CLI. Migration 38 is missing.
