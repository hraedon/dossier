## On the project

dossier's publication-gate story is now structurally complete: three layers of defense (bash regex targeted scan, bash broad scan for LDAP/AD patterns, Python denylist-driven gate with CI secret), plus a read-only history audit and a prepared filter-repo replacements file. The pattern mirrors regista's approach (which was already in regista's uncommitted working tree from a prior session) — the key design decision is that the denylist is a CI secret, not committed, so the identifiers themselves don't leak through the gate definition.

The adcs-lens WI-010 lesson — "the scrub must cover CA CN / all identifier forms, not just hostnames" — is now baked into the publication-review.md scrub instructions and the filter-repo replacements file. The audit found that most of regista's 1651 leaks are from author/committer identity (406 commits × 4 identifiers), which `--replace-text` can't handle — the scrub needs `--name-callback`/`--email-callback` separately. This is documented in the publication-review.

## Gaps to flag

- `scripts/filter-repo-replacements.txt` is gitignored but exists on disk in the repo. It contains real identifiers. If the repo is ever cloned or copied, this file travels with it. Consider storing it outside the repo (e.g., a password manager or a separate gitignored scratch directory).
- The GitHub org name appears in `SUITE.lock:12,18`, `README.md:10,37`, `AGENTS.md:8`, `.github/workflows/ci.yml:32,132-141`, and multiple plan files. If the org name is in scope for the scrub, uncomment the org-name replacement rule in the replacements file. This is a policy decision, not a technical gap.
- The broad scan still doesn't scan `*.json`, `*.toml`, `*.lock` files — only the targeted scan does (after the Kimi fix). The denylist gate scans all text files, so there's coverage, but the broad-scan redundancy is weaker for non-source files.
- The `audit_history_identifiers.py` only scans `+` (added) diff lines. Same-commit add-and-remove is a blind spot. Parent commits eventually catch removed content, but a leak introduced and deleted in a single commit would be missed.
- `_GUARDED_DIRS` now includes `secrets/` but not `.env`, `*.env.local`, or `deploy/winsw/dossier-env.cmd`. These are gitignored but not force-add-protected. The denylist gate's `collect_tracked_paths()` uses `git ls-files`, so force-added files would be caught — but only if the denylist identifiers are set.
