## On the project

dossier's architecture is holding up well under feature addition. The
gateway-as-sole-choke-point pattern (AGENTS.md G1) meant that per-item
integrity (WI-003) was a one-line change in `gateway.py` — regista already
supported `replay(work_item_id=...)`. The two-family auth design (ADR-001)
paid off: the `CredentialBackend` Protocol absorbed the `GroupIdentity` change
without touching the login flow. The `LoginRequired` exception (WI-001/005)
was cleanly separable from the `current_actor` 401 path because the code
already distinguished HTML routes from JSON API routes.

The thing that worries me is the growing gap between "structurally correct" and
"verified against reality" in the LDAP backend. The mock tests now cover 40+
scenarios including group resolution edge cases, but none of them have ever run
against a real AD. The cert-watch lesson (mocked LDAP tests that passed but
were broken on arrival) is documented in Plan 003 but the integration tests
remain skipped. This is the single biggest risk in the codebase.

The throttler (WI-007) is explicitly MVP — in-memory, per-process, per-identifier.
The adversarial review correctly identified that credential stuffing (varying
usernames) is unmitigated and that the fixed-window allows ~4 failures per 300s
without lockout. Both are documented in the class docstring. For a real
deployment, an IP-based rate limiter (or nginx-level throttling) is needed
before this can be considered production-ready.

## On the work done

**Confident in:**
- The `LoginRequired` exception pattern — clean, no JSON leak, handler
  registered once. The `<noscript>` fallback for the review-note toggle was
  the right call (the adversarial reviewer caught that JS-only would block
  non-JS users from submitting review verdicts).
- Per-item integrity replay — regista already had the API; dossier just
  wasn't using it. The test corrupts the InMemoryRegista projection directly,
  which is fragile but proves the badge is per-item.
- scrypt N bump with env-var override — old hashes (N=16384) still verify
  because N is stored per-hash. The `_get_n()` validation (power-of-2, minimum)
  prevents silent misconfiguration.
- mypy --strict — 62 errors fixed, all real type gaps. The `Literal` types on
  `auth_backend` and `group_strategy` enable the `assert_never` pattern.

**Want a second pair of eyes on:**
- The `GroupIdentity` direct-strategy flow does three LDAP searches (user
  lookup, memberOf re-search, group resolution). The third search uses an OR
  filter on all memberOf DNs. If a user is in 50+ groups, this filter could
  exceed AD's filter length limit. No test covers this.
- The `LoginThrottler` concurrency re-check (added from adversarial review)
  narrows the race window but doesn't close it. Two concurrent requests can
  still both pass the re-check before either calls `record_failure`. For a
  true fix, `record_failure` needs to be atomic (check-and-increment under a
  lock). Acceptable for MVP, but documented.
- The `_cn_from_dn` helper uses `ldap3.utils.dn.parse_dn` with a fallback to
  simple split. The fallback doesn't handle escaped commas. The ldap3 path
  should always be available in LDAP deployments, but the fallback exists for
  LocalBackend contexts where ldap3 isn't installed — and local users don't
  have DNs anyway. The fallback is dead code in practice but not harmful.

**Awkward:**
- The parallel agent execution in Phase 1 worked (no file conflicts) but
  Agent B reported "3 pre-existing failures" that were actually Agent A's
  uncommitted changes. Both agents were editing the same working tree. This
  worked because they touched different files, but it could have gone wrong
  if both needed to edit `conftest.py` (which Phase 2 later did).
- Closing work items via `agent-notes work-item close` required applying
  regista migration 035 to the production `dossier` schema on the production Postgres host.
  The migration was in regista's uncommitted working tree (Plan 024) but hadn't
  been applied to the production `regista` database. This was a non-obvious
  blocker — the agent-notes config uses a separate `regista` database
  (different from the `agent_notes` database), and the error message doesn't
  distinguish which database/schema is missing the migration.

## Gaps to flag

- **LDAP integration tests never run against real AD:** `tests/test_ldap_integration.py`
  has 5 tests skipped by default. Until they run, the LDAP backend (including
  the new `GroupIdentity` group resolution) is mock-verified only.
- **Direct group resolution OR filter may exceed AD filter length:** If a user
  is in many groups, the OR filter on all memberOf DNs could hit AD's filter
  length limit. No test covers this. `src/dossier/auth/backends.py`
  `_fetch_direct_groups`.
- **Throttler concurrency race not fully closed:** The re-check narrows the
  window but two concurrent requests can still both pass before either
  increments. `src/dossier/app.py` login route, `src/dossier/auth/throttle.py`.
- **`cast()` calls in gateway.py mask regista API changes:** All regista
  returns are `cast()` because regista has no `py.typed`. If regista changes
  a return shape, the cast will silently accept it. `src/dossier/gateway.py`.
- **Timing side-channel on locked login:** When `is_locked` returns True, the
  route returns 429 before the constant-time dummy hash runs, leaking lockout
  state via timing. `src/dossier/app.py` login route.
- **`test_integrity_check_is_per_work_item` mutates private InMemoryRegista
  state:** `gw._reg._work_items[id_a]["current_state"] = "done"` reaches into
  the in-memory store's private dict. Will break if the representation changes.
  `tests/test_ui.py`.
- **regista migration 035 was manually applied to production:** The migration
  (`035_event_chain_head_genesis_sentinel.sql`) is in regista's uncommitted
  working tree. I applied it directly via SQL to the `dossier` schema on
  the production Postgres host because the `regista_app` user didn't have migration-runner
  permissions (or the `run_migrations` function wasn't discovering it). The
  regista team should verify this migration is properly tracked and applied
  to all schemas.
