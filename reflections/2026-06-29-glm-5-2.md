## On the project

The auth layer is where provenance lives or dies — G1 (attribution) is only
as good as the `stable_id` the backend produces. Plan 003's insistence on
`objectGUID` over `sAMAccountName` or DN is the right call: it's the same
immutability argument as keying on a UUID for local users, and it survives
the real-world AD operations (rename, OU move) that break the alternatives.
The `Principal → Actor` pipeline through `principal_to_actor` is clean — one
keystone, no client-input path, every backend converges on it.

The two-family auth design (credential vs federated) documented in the ADR
is a good architectural seam. The guardrails are cheap (a rename, a helper
extraction, a doc page) and they prevent the scenario where adding Entra
OIDC later requires touching the LDAP or Local credential path. The key
insight — that `_credential_login` isolates the "password arrives at /login"
assumption — means a future `/auth/callback` route is a sibling, not a
modification.

## On the work done

Confident in: the search-then-bind flow itself (service account searches,
user DN re-binds, bind result checked), the `objectGUID` byte-to-UUID
conversion (tested with round-trip, invalid bytes, string normalization),
the LDAPS+CA pinning (CERT_REQUIRED always, never validate=NONE), and the
`try/finally` connection cleanup. The 33 mocked tests cover the key paths:
success, wrong password, user not found, empty password, service bind
failure, no objectGUID, LDAP exception, and the security test proving groups
aren't fetched on wrong password.

Want a second pair of eyes on: the `_credential_login` helper's JSON error
handling. I added a try/except around `request.json()` that returns `None, form_req`
on any exception, and a `isinstance(payload, dict)` check. This prevents
a 500 on malformed JSON, but it means a malformed JSON body is treated
as "invalid credentials" rather than a 400 Bad Request. That's defensible
(don't leak parsing details) but it's a behavior choice worth confirming.

Awkward: the mock-based testing approach works but has a fundamental
limitation — the mocks verify that the code calls `ldap3.Connection` with
the right kwargs, but they can't verify that the real ldap3 library behaves
the way the mock simulates. The plan explicitly calls this out ("run them
and watch one fail before trusting them — the cert-watch broken-on-arrival
LDAP test is the cautionary tale"). The integration test scaffolding exists
but hasn't been run against the real AD yet. Until it has, the LDAP backend
is structurally correct but unproven against a live directory.

## On what remains

Needed before LDAP can be trusted in production:
- **Run the integration tests against the real AD.** The scaffolding
  is in `tests/test_ldap_integration.py` — it needs the `svc-bind`
  credential and a test user account. This is the single most important
  remaining step. The cert-watch lesson is explicit: mocked tests alone are
  not sufficient for LDAP.
- **WI-008: Group identities as GUID+name, not DNs.** Currently groups are
  returned as DNs, which break on OU rename. The plan says "prefer
  objectGUID + name" — this was deferred for MVP but Plan 004 (teams) will
  need it.
- **WI-009: Enforce CA pinning in strict mode.** Currently `strict=True`
  requires LDAPS but not `DOSSIER_LDAP_CA_CERT_FILE`. A deployment could
  silently fall back to the system trust store.

Nice to have:
- The `_fetch_nested_groups` method returns DNs — Plan 004 may want
  GUIDs. The `tokenGroups` alternative mentioned in the plan was not
  implemented; the `LDAP_MATCHING_RULE_IN_CHAIN` approach was chosen
  instead (simpler, gives DNs directly). Documented as a design choice
  in the code comment but not in the ADR.
- Login throttling (WI-007 from the existing backlog) is still open —
  the LDAP backend doesn't add any throttling, and AD lockout policy is
  a directory-side concern, but dossier should at least log repeated
  failures.

## Gaps to flag

- **Integration tests never run against real AD:** `tests/test_ldap_integration.py`
  has 5 tests that are skipped by default. They need `DOSSIER_LDAP_E2E=1` +
  env vars + the `svc-bind` credential. Until they run and pass, the
  LDAP backend is structurally sound but unverified. The mock tests prove
  the logic; they don't prove the ldap3 API assumptions.
- **Groups are DNs, not GUID+name (WI-008):** `src/dossier/auth/backends.py`
  `_fetch_nested_groups` and the direct `memberOf` path both return DN
  strings. Plan 003 WI-3 says "prefer group objectGUID + name." This is a
  deferred improvement, not a bug, but Plan 004 (teams) will need stable
  group identifiers.
- **Strict mode doesn't enforce CA pinning (WI-009):** `src/dossier/config.py`
  `load_ldap_config(strict=True)` requires `ldaps://` but not
  `DOSSIER_LDAP_CA_CERT_FILE`. A deployment could fall back to the system
  trust store without realizing it. The constructor logs a warning, but
  strict mode should arguably fail-closed.
- **`read_only=True` on service connection is mock-verified only:** The
  `read_only=True` kwarg is passed to the mock `ldap3.Connection` but has
  never been tested against a real AD. If ldap3 doesn't enforce it (or the
  AD ignores it), the least-privilege posture is weaker than assumed.
- **ruff reformatted three pre-existing test files:** `test_deferred.py`,
  `test_e2e_postgres.py`, `test_gateway.py` were reformatted by `ruff
  format` during this session. The changes are purely cosmetic (line
  wrapping) but they inflate the diff. They're in the working tree and will
  be committed with the Plan 003 changes.
