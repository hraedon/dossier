## On the project

The workflow-derived UI pattern continues to pay off. Adding a state + four
transitions to the YAML automatically surfaced them in the detail page's
transition dropdown ŌĆö zero code change needed beyond adding labels to the
lookup dicts. The `status_pill_class` and filter dropdown needed one line each.
This is the right architecture for a tracker whose state model is the contract.

The `deferred` / `blocked` distinction is genuinely worth the contract change.
Collapsing them would throw away signal ŌĆö "I chose to stop" vs "I can't
proceed" are different facts, and a provenance instrument exists to preserve
exactly that kind of distinction. The plan's rationale holds up under
implementation.

One thing that surprised me: `defer` and `resume` are authorship transitions
(not in `_NON_AUTHOR_TRANSITIONS`), which means an actor who defers an item
becomes an author and can't later review it. This is the correct security
posture ŌĆö parking/re-engaging is an active triage decision ŌĆö but it was
implicit, an emergent property of the default rather than a deliberate
decision. The kimi reviewer caught this, and I documented it in the validators
docstring and added a test proving the behavior. It should be fine, but it's
the kind of thing that deserves to be explicit, not assumed.
