## On the project

dossier is reaching the point where it's deployable as a suite component rather than a
standalone app. The Plan 013 work makes it read the shared `REGISTA_*` config vocabulary,
ship as a container image with a pinned regista SHA, report health in the suite-conformant
shape, and install as a Windows Service. The architecture is holding up well — dossier
remains a thin face over regista, and the new packaging/provisioning surface doesn't
violate the "no work-state in dossier" rule.

The thing that strikes me is how much of the value is in the *seams*: the `_resolve_env`
function that bridges legacy `DOSSIER_*` to canonical `REGISTA_*` vars, the `SUITE.lock`
that pins face-to-spine versions, the `_check_provisioned` that gives operators an
actionable error instead of a stack trace. None of these are complex, but they're what
make the suite deployable rather than just runnable.

## Gaps to flag

- **health.py:41** — `gw.integrity()` runs a full per-project replay on every `/healthz`
  call. For a production deployment with many work items, this could be expensive. Consider
  caching or sampling.
- **cli.py:_check_provisioned** — opens a separate psycopg connection that doesn't go
  through regista's `ConnectionManager`. If the DSN has special parameters (sslmode,
  connect_timeout), they won't be applied consistently. The `require_ssl` parameter is now
  used but other params are not.
- **Dockerfile:HEALTHCHECK** — the healthcheck command is a complex inline Python one-liner.
  If the JSON parsing fails (malformed response), it exits 1 (healthy) rather than 0
  (unhealthy), which is the wrong default for a healthcheck. Actually wait — `exit 1`
  means *unhealthy*. So this is correct. But the inline Python is fragile.
- **CI:SUITE.lock parsing** — `grep 'ref:' SUITE.lock | head -1 | awk '{print $2}'` will
  break if the YAML is quoted or reformatted. Both reviewers flagged this. Should use a
  proper YAML parser.
- **deploy/winsw/install.ps1** — still runs the service as SYSTEM by default. A dedicated
  low-privilege account would be better, but creating Windows service accounts requires
  domain admin privileges that the install script can't assume. Documented as a limitation.
- **docker-compose.yml:43** — the dev session secret `dev-only-not-for-production-use-...`
  is committed and could be accidentally used in production. It's clearly labeled but a
  stronger pattern would be to require `.env` and fail if the secret is missing.
