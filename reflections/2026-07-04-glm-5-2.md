## On the project

The gap between "v1 stubs knowingly" and "deployed-as-is provenance is broken" is the core tension in dossier right now. The stubs were documented — the TODO comments clearly said "this is NOT a valid Ed25519 public key" — but documentation doesn't prevent deployment. The fix closes the most dangerous gap: the keys are now cryptographically valid. But the deeper architectural gap remains: these Ed25519 keys are registered with regista's principal registry but not wired to the event signing path (HMAC-SHA256 is still the active mechanism; Ed25519 signing is Plan 026). So the keys are valid but dormant. That's the right sequencing — get the UX and key lifecycle right first, then flip the signing mechanism — but it means "provenance is fixed" is only half true.

The adversarial review process is working well. Two reviewers with different model lineages found overlapping but distinct issues. The Nemotron reviewer was more security-focused (path traversal, symlink attacks, log injection); the Kimi reviewer was more architecture-focused (keyset file not updated, cross-project atomicity, dead keys). Both flagged the fake dual-control and the partial-failure-across-projects problem. The cross-lineage review pattern — where I fix issues found by one lineage and then run a second lineage to catch what the first missed — is valuable.

## On the work done

I'm confident in the core fix: `generate_ed25519_keypair()` using PyNaCl is correct, the `PrincipalKeyManager` storage is hardened (atomic write, path validation, 0600 perms, O_NOFOLLOW), and the break-glass action now does something real. The tests do proper round-trip verification (sign with private key from disk, verify with public key from registry) — the initial tests were flawed (only proved wrong keys fail, not that correct keys work).

The adversarial review hardening was the most valuable part. The initial implementation had a path traversal vulnerability in `principal_id` that I would not have caught on my own — I was focused on the cryptographic correctness and missed the filesystem attack surface. The `O_NOFOLLOW` and atomic-write fixes were also things I wouldn't have thought of. The `assert` → `RuntimeError` change is a good defensive practice I should internalize.

What I'm less confident in: the break-glass "dual-control" is still attestation-only (the confirmer doesn't authenticate). I documented this more clearly in the code comment, but it's still a security control that doesn't control. The partial-failure-across-projects issue is logged but not surfaced to the user — the admin sees a 303 redirect even if 2 of 3 projects failed. Both are acknowledged v1 limitations, but they're the kind of things that bite in production.

## On what remains

1. Wire the generated Ed25519 keys to the actual signing path — update regista's keyset file with per-principal entries so events are signed with the actor's key, not the project HMAC key (Plan 026)
2. Implement real two-step break-glass dual-control (v1.5) — the confirmer must approve via their own authenticated session
3. Surface partial-failure errors to the user — currently the admin sees a success redirect even when some projects failed

## Gaps to flag

- `app.py:1011-1020` — Break-glass dual-control is attestation-only, not authentication. The confirmer_id is checked against the admin set but the confirmer never approves in their own session. Documented as v1.5 but is a real security gap if deployed as-is.
- `app.py:771-790, 896-914, 1001-1040` — All key lifecycle actions (rotation, enrollment, break-glass) loop over projects with no transaction. Partial failures are logged but not surfaced to the user. A principal can end up with a new key in project A and the old key in project B.
- `keys.py` — `PrincipalKeyManager` writes private keys to disk but never cleans up old keys on rotation. Superseded/revoked private keys accumulate, defeating forward secrecy if the filesystem is compromised.
- `src/dossier/principals.py` — `InMemoryPrincipalKeyStore` is a test double in production source code. Could be accidentally used in production.
- `app.py` — No rate limiting on `/me/key/rotate`, `/admin/principals/enroll`, `/admin/break-glass`. An authenticated user can spam key rotations, filling the key directory and registry.
- `keys.py` — The generated Ed25519 keys are registered with regista's principal registry but NOT added to the HMAC keyset file. Events are still signed with the project-level HMAC key, not the per-actor Ed25519 key. This is by design (Plan 026), but means "provenance is fixed" is only half true — the keys are valid but dormant.
